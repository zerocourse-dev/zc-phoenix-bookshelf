# Bookshelf

**ZeroCourse — Elixir & Phoenix, Course 5: Phoenix Framework**

A Phoenix app the way you'd meet one at work: the frame is up — router,
contexts, schemas, migrations, JSON views, channel socket, fallback
controller — and the load-bearing pieces are yours to build. If you know
Rails, every file here has a cousin you already understand; the tests
pin down where Phoenix thinks differently.

## What you build

Three graded surfaces (grep for `GRADED` and `NotImplementedError`):

| Surface | Where | What the tests demand |
|---------|-------|----------------------|
| Query composition | `Bookshelf.Library.list_books/1` | One composable Ecto query: status/author/year filters + three sort orders. No filtering in Elixir after `Repo.all` |
| Write path + events | `Library.add_review/2`, `Review.changeset/2`, `Library.book_stats/1` | Changeset validation (rating 1..5), a unique index surfaced as a changeset error, a PubSub broadcast on success, and DB-side aggregates |
| The edges | `BookshelfWeb.Api.BookController`, `BookshelfWeb.ActivityChannel` | A JSON API with uniform error rendering via `action_fallback`, and a channel that pushes `review_added` to every connected client |

The HTML scaffold under `lib/bookshelf_web/controllers/` and everything
else generated by `phx.new` is working scaffolding — read it, don't fear
it.

## Getting started

You need PostgreSQL running locally (any superuser `postgres`/`postgres`
setup matches `config/dev.exs` and `config/test.exs`).

```bash
mix deps.get
mix ecto.setup
mix test          # 29 failures to turn green
mix phx.server    # http://localhost:4000
```

Run one surface at a time:

```bash
mix test test/bookshelf/library_test.exs
mix test test/bookshelf_web/controllers/api/book_controller_test.exs
mix test test/bookshelf_web/channels/activity_channel_test.exs
```

Five scaffolding smoke tests pass from the start; the 29 graded tests
fail until you implement the stubs.

## The checkpoint, beyond this repo

This repo is the CI-graded core. The full checkpoint also asks you to:

1. **Add authentication with `mix phx.gen.auth`** — run the generator
   yourself and wire the Books UI behind it. That is the real workflow;
   generated auth is something you configure, not something you download.
2. **Deploy to Fly.io** and note what `fly launch` had to know about
   your app.
3. **Write the Rails comparison** — what was faster, slower, better,
   worse than the equivalent Rails app you've built. Submit it with
   your PR description; your mentor reviews it with the code.

## Tips

- `list_books/1` wants `Enum.reduce/3` over the filters, building the
  query up clause by clause. `exclude(query, :order_by)` is how a sort
  option replaces the default ordering.
- `avg/1` comes back as a `Decimal` — see `Decimal.to_float/1`.
- The broadcast goes through `BookshelfWeb.Endpoint.broadcast/3`; the
  channel test subscribes with `subscribe_and_join/3` and asserts with
  `assert_push`.
- `unique_constraint/3` needs the index name the migration created:
  `:reviews_book_id_reviewer_index`.
- The FallbackController already renders `{:error, %Ecto.Changeset{}}`
  as 422 and `{:error, :not_found}` as 404 — your actions just return
  those tuples.
